# Dynamic dispatch: flock + pbsdsh (dynamic-dispatch.md)

When you have **many short tasks** (e.g. dozens of `mlmm dft`
single points), submitting one PBS job per task wastes scheduler time
and produces noisy queue activity. Better: submit **one** multi-node
PBS job that grabs N nodes, runs one worker per node, and the workers
pull tasks from a shared list with file-lock-protected counter
increment.

This is the same pattern used by `pbsdsh` + `flock` on Torque/PBSPro
clusters. The recipe below is **placeholder-driven** — fill in
`<NCPU>`, `<NGPU>`, `<QUEUE>`, `<CUDA_MODULE>`, `<YOUR_ENV>`,
`<TASK_LIST_FILE>` from `mlmm-env-detect/SKILL.md`.

## Files involved

```
dispatcher.sh           # PBS script — qsub this
_worker.sh              # generated by dispatcher, runs on each node
<TASK_LIST_FILE>        # one task per line (e.g. directory paths)
_state.txt              # current task counter (auto-managed)
_lock                   # flock file (auto-managed)
```

## dispatcher.sh

```bash
#!/usr/bin/env bash
#PBS -N mlmm_dispatch
#PBS -q <YOUR_QUEUE>
#PBS -l select=<N_NODES>:ncpus=<NCPU>:ngpus=<NGPU>:mem=<MEM>gb,walltime=<HH:MM:SS>
#PBS -j oe

cd "${PBS_O_WORKDIR}"

ROOT_DIR="$PWD"
TASK_LIST="${ROOT_DIR}/<TASK_LIST_FILE>"
STATE_FILE="${ROOT_DIR}/_state.txt"
LOCK_FILE="${ROOT_DIR}/_lock"
WORKER="${ROOT_DIR}/_worker.sh"

# Initialize state
echo 1 > "${STATE_FILE}"
TOTAL_TASKS=$(wc -l < "${TASK_LIST}")

# Generate the worker (heredoc, with placeholder env)
cat > "${WORKER}" <<'WORKER_EOF'
#!/usr/bin/env bash
ROOT_DIR="$1"
TOTAL_TASKS="$2"
STATE_FILE="$3"
LOCK_FILE="$4"
TASK_LIST="$5"

# Env (replace <CUDA_MODULE> and <YOUR_ENV> via sed below if needed)
command -v module >/dev/null && module load <CUDA_MODULE>
source "$(conda info --base)/etc/profile.d/conda.sh"
conda activate <YOUR_ENV>
export PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True

while true; do
    task_num=""

    # Atomic counter increment under flock
    {
        flock -x 200
        if [[ -f "${STATE_FILE}" ]]; then
            current=$(cat "${STATE_FILE}")
            if (( current <= TOTAL_TASKS )); then
                task_num="${current}"
                echo $(( current + 1 )) > "${STATE_FILE}"
            fi
        fi
    } 200>"${LOCK_FILE}"

    [[ -z "${task_num}" ]] && break

    rel_path=$(sed -n "${task_num}p" "${TASK_LIST}")
    [[ -z "${rel_path}" ]] && continue

    task_dir="${ROOT_DIR}/${rel_path}"

    echo "[$(date +'%F %T')] $(hostname): START ${rel_path}"

    if ! (
        cd "${task_dir}"
        bash run.sh
    ); then
        echo "[$(date +'%F %T')] $(hostname): ERROR ${rel_path}" >&2
        continue
    fi

    echo "[$(date +'%F %T')] $(hostname): DONE  ${rel_path}"
done
WORKER_EOF

chmod +x "${WORKER}"

# Launch one worker per allocated vnode
if [[ -n "${PBS_NODEFILE:-}" ]]; then
    mapfile -t vnode_indices < <(awk '!seen[$1]++ {print NR-1}' "${PBS_NODEFILE}")
    pids=()
    for idx in "${vnode_indices[@]}"; do
        pbsdsh -n "${idx}" -- \
            env PBS_NODEFILE="${PBS_NODEFILE}" \
                "${WORKER}" "${ROOT_DIR}" "${TOTAL_TASKS}" "${STATE_FILE}" "${LOCK_FILE}" "${TASK_LIST}" &
        pids+=("$!")
    done
    for pid in "${pids[@]}"; do wait "${pid}"; done
else
    # Fallback for non-PBS environments
    "${WORKER}" "${ROOT_DIR}" "${TOTAL_TASKS}" "${STATE_FILE}" "${LOCK_FILE}" "${TASK_LIST}"
fi

# Cleanup
rm -f "${WORKER}" "${STATE_FILE}" "${LOCK_FILE}"
echo "All ${TOTAL_TASKS} tasks attempted."
```

## Task list format

`<TASK_LIST_FILE>` is one line per task; each line is a relative path
to a directory containing a `run.sh` (or whatever your worker
expects):

```
dft_R/
dft_TS1/
dft_IM/
dft_TS2/
dft_P/
seg_03/dft_R/
seg_03/dft_TS/
...
```

Each task directory has its own `run.sh` that the worker invokes via
`(cd "${task_dir}" && bash run.sh)`.

## SLURM analog

The SLURM equivalent uses `srun --multi-prog` or per-task `sbatch
--array=...` with the same flock-based counter. The skeleton:

```bash
#SBATCH --job-name=mlmm_array
#SBATCH --array=1-<N_TASKS>%<N_NODES>
#SBATCH ...

cd "${SLURM_SUBMIT_DIR}"
rel_path=$(sed -n "${SLURM_ARRAY_TASK_ID}p" <TASK_LIST_FILE>)
cd "${rel_path}" && bash run.sh
```

Array jobs (`--array`) are simpler than the pbsdsh-based dispatcher
because SLURM does the load-balancing and counter tracking.

## When to use this pattern

- 50+ short tasks, each < 1 h.
- Tasks are independent (no inter-task data flow).
- You want to amortize one queue submission across many compute
  nodes.

When **not** to use:

- 1–5 long tasks (just submit them individually).
- Tasks have inter-dependencies (use a workflow engine — Nextflow,
  Snakemake, etc., not this pattern).
